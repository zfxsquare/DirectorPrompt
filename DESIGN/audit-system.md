# 审计系统

## 概述

审计系统守护叙事一致性。它不生成叙事, 不修改状态, 不更新记忆 — 只做一件事: 判断 Narrator 生成的叙事文本是否与已知的规则、状态、人物、记忆一致。

## 核心原则: 上下文隔离

审计的可靠性来自上下文隔离, 不是上下文丰富。过长的上下文会导致 AI 注意力分散, 无法发现细节问题。

### 输入隔离

每次审计, Audit Agent 只拿到:
- **Narrator 输出的叙事正文**

不给 Narrator 的思考过程 — 审计不应知道作者的内心活动, 且 AI 可能把思考中的错误当作正确。不给其他系统的完整数据 — 那些由 Audit Agent 自己按需查询。

### 分维度隔离

每个审计维度是一个独立的子审计, 上下文只有"叙事正文 + 本维度的校验说明"。维度之间互不干扰, 各自只关注自己负责的方面。

## 审计维度

| 维度 | 校验内容 | 工具 |
|------|---------|------|
| 设定一致性 | 叙事是否违反世界设定 | query_knowledge |
| 状态一致性 | 叙事中的状态描述与当前状态值是否矛盾 | get_all_state, get_character_state |
| 人物一致性 | 人物行为/存在是否合理 | get_scene_characters, get_character, get_relations |
| 时间一致性 | 时间描述是否矛盾 | query_scene |
| 记忆一致性 | 叙事是否与已发生的事件矛盾 | query_memory |

### 关于人物一致性的边界

人物性格突变是否算违规, 是语义判断, 不是硬规则。"沉稳的人突然暴怒"可能合理 (有叙事理由), 也可能不合理 (AI 幻觉导致的性格漂移)。Audit Agent 用 LLM 结合人物描述和叙事上下文做判断。

## 审计流程

### 第一阶段: 分维度并行审计

所有维度并行执行, 每个维度独立调用 Audit Agent:

```
审计启动
    │
    ├─ 维度1: 设定一致性审计
    │     上下文: 叙事正文 + "校验叙事是否违反世界设定"
    │     工具: query_knowledge
    │     → 发现问题 → add_violation(type, description, severity, suggestion)
    │
    ├─ 维度2: 状态一致性审计
    │     上下文: 叙事正文 + "校验叙事中的状态描述是否与当前状态值一致"
    │     工具: get_all_state, get_character_state
    │     → 发现问题 → add_violation(...)
    │
    ├─ 维度3: 人物一致性审计
    │     上下文: 叙事正文 + "校验人物行为/存在是否合理"
    │     工具: get_scene_characters, get_character, get_relations
    │     → 发现问题 → add_violation(...)
    │
    ├─ 维度4: 时间一致性审计
    │     上下文: 叙事正文 + "校验时间描述是否矛盾"
    │     工具: query_scene (查前序场景的 timeLabel)
    │     → 发现问题 → add_violation(...)
    │
    └─ 维度5: 记忆一致性审计
          上下文: 叙事正文 + "校验叙事是否与已发生的事件矛盾"
          工具: query_memory
          → 发现问题 → add_violation(...)

    全部并行执行
```

每个维度通过 `add_violation` tool call 往一个共享列表里追加问题。

### 第二阶段: 去重合并

所有维度并行完成后, 调用一次 Audit Agent 做最终处理:

```
输入: 所有维度收集到的问题列表
任务: 去重 + 合并相似问题
输出: 精简后的问题列表
```

### 严重程度与代码过滤

三级严重程度:

| 级别 | 含义 | 代码处理 |
|------|------|---------|
| unacceptable | 不可接受, 逻辑硬伤 | 阻断或标记 |
| severe | 严重矛盾 | 阻断或标记 |
| general | 一般, 轻微瑕疵 | **代码层过滤掉, 不返回给流程** |

`general` 级别的问题在代码层面被过滤掉, Audit Agent 自身不会看到它们存在 — 返回给流程的结果等同于"审计通过"。这避免"审计模型天生挑刺"导致的过度发现问题, 代码层过滤掉低级别噪音。

## 问题输出

```json
{
    "violations": [
        {
            "type": "setting",
            "description": "叙事中描述张三使用火魔法, 但世界设定中张三只会水系魔法",
            "severity": "unacceptable",
            "suggestion": "改为水系或冰系魔法, 或修改叙事使张三借助了外部火源"
        },
        {
            "type": "character",
            "description": "李四在第二场景已经死亡, 但本场景中李四再次出现并与主角对话",
            "severity": "unacceptable",
            "suggestion": "移除李四的出场, 或改为幻觉/回忆"
        },
        {
            "type": "time",
            "description": "上一场景为'第一天深夜', 当前场景timeLabel为'第二天凌晨', 但叙事中提到'三天后的早晨'",
            "severity": "severe",
            "suggestion": "修正时间描述为'第二天凌晨'"
        }
    ]
}
```

## 审计结果处理模式

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| 阻断模式 | 不通过则反馈给 Narrator 重生成, 设最大重试次数 | 精修阶段, 追求质量 |
| 标记模式 | 不通过但放行, UI 上标记警告 | 草稿阶段, 追求速度 |
| 关闭 | 不做审计 | 快速迭代或用户信任 AI |

### 阻断模式的反馈循环

```
Audit 不通过 (存在 unacceptable 或 severe 问题)
    │
    ▼
将审计报告 (violations + suggestions) 作为附加约束注入 Narrator
    │
    ▼
Narrator 重新生成 (附带"上一次的问题"和"修改建议")
    │
    ▼
再次审计
    │
    ├─ 通过 → 放行
    ├─ 不通过, 重试次数 < 上限 → 再次反馈重生成
    └─ 不通过, 重试次数 = 上限 → 标记警告放行, 避免死循环
```

最大重试次数可配置, 默认 2 次。超过后即使仍有问题也放行, 但 UI 标记"审计未通过"。

### 审计与状态/记忆更新的时序

审计不通过时, 状态和记忆的更新不应该执行 — 因为叙事文本会被丢弃重生成:

```
Narrator 生成叙事
    │
    ▼
Audit Agent 审计
    │
    ├─ 通过或标记模式放行
    │       │
    │       ▼
    │   State Agent 提取状态变更
    │   Memory Sub-Agent 更新记忆和人物
    │       │
    │       ▼
    │   写入数据库, 更新 UI
    │
    └─ 阻断模式不通过
            │
            ▼
        反馈 Narrator 重生成 (状态和记忆不更新)
```

## 模型选择

Audit Agent 适合低温中小模型:

- **低温** (temperature 0-0.3): 审计需要确定性判断, 不需要创造性
- **中小模型**: 判别式判断 ("是否一致") 比生成式任务 ("写叙事") 简单
- **结构化输出**: 用 JSON Schema 约束输出格式, 确保可解析

## 配置

```
AuditConfig {
    mode: enum              // blocking / marking / disabled
    maxRetries: int         // 阻断模式最大重试次数, 默认 2
    dimensions: string[]    // 启用哪些校验维度, 默认全部
}
```

`dimensions` 让用户可以按需关闭某些维度 — 比如"我不关心时间一致性, 关掉"。

## 审计在流水线中的位置

```
导演输入
    │
    ▼
Orchestrator 编排:
    ┌─ Knowledge Agent 检索知识 ──────┐
    ├─ Memory Sub-Agent 召回记忆 ─────┤ (并行)
    ├─ 系统注入当前状态 ──────────────┤
    ├─ 系统注入人物 + 关系网络 ────────┤
    └─ 系统注入场景时间信息 ──────────┘
                │
                ▼
         Narrator 生成叙事
                │
                ▼
         Audit Agent 审计 ──── 不通过(阻断) ──→ 反馈重生成 ↑ (maxRetries 次)
                │
            通过/放行
                │
         ┌──────┴──────┐
         ▼             ▼
   State Agent    Memory Sub-Agent
   提取状态变更    更新记忆 + 人物
         │             │
         └──────┬──────┘
                ▼
         写入数据库 + 更新 UI
```

审计位于 Narrator 之后、状态/记忆更新之前。只有通过审计的叙事才会被持久化。
